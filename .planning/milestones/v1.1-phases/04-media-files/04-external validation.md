# Phase 4 External Validation

## Summary

Phase 4 is substantial and mostly solid: media search filtering, metadata extraction, download, and upload flows are implemented, the codebase builds, and the full test suite passes. However, several important gaps remain between the current implementation, the formal requirements, and the strength of the verification report.

## Verification Snapshot

- `npm test`: 314/314 tests passing
- `npx tsc --noEmit`: passing
- The command surface is present and non-trivial
- The remaining concerns are primarily about requirement coverage and response correctness

## Findings

### 1. `READ-05` is not fully implemented against the requirement

**Status:** Open  
**Severity:** Critical

### Problem

The requirement in `REQUIREMENTS.md` says users can filter with **any of the 17 MTProto search filters**, but the implementation currently exposes only 8 filters while the phase verification report still marks the requirement as satisfied.

### Code snippet

```ts
export const FILTER_MAP: Record<string, () => InstanceType<any>> = {
  photos: () => new Api.InputMessagesFilterPhotos(),
  videos: () => new Api.InputMessagesFilterVideo(),
  documents: () => new Api.InputMessagesFilterDocument(),
  urls: () => new Api.InputMessagesFilterUrl(),
  voice: () => new Api.InputMessagesFilterVoice(),
  music: () => new Api.InputMessagesFilterMusic(),
  gifs: () => new Api.InputMessagesFilterGif(),
  round: () => new Api.InputMessagesFilterRoundVideo(),
};
```

### Why this is a problem

- The implementation does not match the formal requirement as written.
- Verification language currently overstates completion.
- This is a real scope gap, not just a test gap.

### Possible solution methods

- Implement the remaining MTProto search filters and expose them through `FILTER_MAP`.
- Or explicitly narrow the requirement and update planning/verification artifacts to match the implemented scope.

### 2. Album send can silently return a partial album

**Status:** Open  
**Severity:** Important

### Problem

Album send assumes message IDs are sequential, re-fetches a contiguous range, and treats any non-empty subset as success.

### Code snippet

```ts
if (isAlbum) {
  const ids = Array.from(
    { length: files.length },
    (_, i) => (result as any).id - files.length + 1 + i,
  );
  const albumMsgs = await client.getMessages(entity, { ids });
  const validMsgs = albumMsgs.filter(Boolean);

  if (validMsgs.length > 0) {
    const serialized = validMsgs.map(m => serializeMessage(m as any));
    outputSuccess({ messages: serialized, sent: serialized.length });
  } else {
    const serialized = serializeMessage(result as any);
    outputSuccess({ messages: [serialized], sent: 1 });
  }
}
```

### Why this is a problem

- If Telegram returns gaps or non-sequential IDs, the command can under-report the album.
- The command can still look successful while omitting some sent items.
- The current verification wording is stronger than the actual safety of this strategy.

### Possible solution methods

- Detect partial album re-fetch and treat it as degraded/fallback behavior explicitly.
- Return a warning/flag when only a subset is recovered.
- Use a more reliable grouping/re-fetch strategy if the API allows it.

### 3. Media size extraction does not follow the intended BigInteger conversion strategy

**Status:** Open  
**Severity:** Important

### Problem

The phase plan called for `.toJSNumber()` handling for Telegram numeric values, but the implementation uses `Number(...)` directly.

### Code snippets

```ts
return {
  filename: null,
  fileSize: bestSize?.size != null ? Number(bestSize.size) : null,
  mimeType: 'image/jpeg',
  width: bestSize?.w ?? null,
  height: bestSize?.h ?? null,
  duration: null,
};
```

```ts
return {
  filename,
  fileSize: doc.size != null ? Number(doc.size) : null,
  mimeType: doc.mimeType ?? null,
  width,
  height,
  duration,
};
```

### Why this is a problem

- If gramjs returns big-integer objects here, `Number(...)` can yield `NaN` or incorrect values.
- That affects JSON metadata, human-readable annotations, and download result sizes.

### Possible solution methods

- Convert big-integer values explicitly with `.toJSNumber()` when available.
- Add tests that simulate gramjs big-integer-like values for media sizes.

### 4. Single-download `-o` can return inconsistent `path` and `filename`

**Status:** Open  
**Severity:** Important

### Problem

When the user passes `-o` for a single download, the file is written to the override path, but the returned `filename` still reflects the Telegram/autogenerated media name.

### Code snippet

```ts
const filename = rawFilename
  ? basename(rawFilename)
  : generateFilename(mediaType, msgId, mediaInfo?.mimeType ?? null);

let targetPath: string;
if (opts.output) {
  if (isBatch) {
    targetPath = resolve(opts.output, filename);
  } else {
    targetPath = resolve(opts.output);
  }
} else {
  targetPath = resolve(process.cwd(), filename);
}

results.push({
  path: targetPath,
  filename,
  size: mediaInfo?.fileSize ?? 0,
  mediaType,
  messageId: msgId,
});
```

### Why this is a problem

- `path` and `filename` can describe different filenames.
- This makes the `DownloadResult` contract ambiguous for humans and agents.

### Possible solution methods

- For single-download `-o`, derive `filename` from the final saved path.
- Or return both `savedFilename` and `originalFilename` explicitly.

### 5. Upload type verification is stronger than the actual implementation

**Status:** Open  
**Severity:** Important

### Problem

The verification story implies strong upload type enforcement (`jpg=photo`, `mp4=video`, `ogg=voice`), but the implementation only explicitly steers `voice` and `document`. Photo/video mostly rely on gramjs defaults.

### Code snippet

```ts
if (!isAlbum) {
  const ext = extname(files[0]);
  const fileType = detectFileType(ext);

  if (fileType === 'voice') {
    sendParams.voiceNote = true;
  } else if (fileType === 'document') {
    sendParams.forceDocument = true;
  }
}
```

### Why this is a problem

- The code does not itself strongly prove the documented type behavior for all file classes.
- The current verification language is stronger than the enforcement logic.

### Possible solution methods

- Either explicitly enforce/send type hints for photo/video too, if needed.
- Or weaken the wording to say that type detection mainly affects voice/document handling while photo/video follow gramjs defaults.

## Recommended Next Fix Iteration

1. Decide whether `READ-05` should mean all 17 filters or the implemented 8-filter subset.
2. Make album send behavior explicit and safe when the re-fetch returns only a subset.
3. Fix media size conversion for gramjs numeric objects.
4. Align `DownloadResult.filename` with the actual saved file path when `-o` is used.
5. Tighten verification wording or implementation around upload type behavior.
