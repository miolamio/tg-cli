<task type="auto" tdd="true">
  <name>Task 2: Core library modules -- types, output, errors, config, prompt</name>
  <files>src/lib/types.ts, src/lib/output.ts, src/lib/errors.ts, src/lib/config.ts, src/lib/prompt.ts, tests/unit/output.test.ts, tests/unit/config.test.ts</files>
  <behavior>
    - outputSuccess({ user: "test" }) writes `{"ok":true,"data":{"user":"test"}}` + newline to stdout, nothing to stderr
    - outputError("fail", "AUTH_ERR") writes `{"ok":false,"error":"fail","code":"AUTH_ERR"}` + newline to stdout
    - outputError("fail") writes `{"ok":false,"error":"fail"}` (no code field) to stdout
    - logStatus("connecting...") writes "connecting..." + newline to stderr, nothing to stdout
    - logStatus("msg", true) writes nothing (quiet mode)
    - resolveCredentials returns { apiId, apiHash } when TG_API_ID and TG_API_HASH env vars are set
    - resolveCredentials returns { apiId, apiHash } from config file when env vars not set
    - resolveCredentials returns null when neither env vars nor config have credentials
    - Config file is created at XDG path via conf with projectName "tg-cli"
    - TgError, CredentialError, SessionError are proper Error subclasses
  </behavior>
  <action>
1. Create `src/lib/types.ts`:
   - `GlobalOptions` interface: `{ json: boolean, human: boolean, verbose: boolean, quiet: boolean, profile: string, config?: string }`
   - `ProfileData` interface: `{ session: string, phone?: string, created?: string }`
   - `TgConfig` interface: `{ apiId?: number, apiHash?: string, profiles: Record<string, ProfileData> }`
   - `SuccessEnvelope<T>` type: `{ ok: true, data: T }`
   - `ErrorEnvelope` type: `{ ok: false, error: string, code?: string }`
   - `OutputEnvelope<T>` union type of both
   - Export all types

2. Create `src/lib/output.ts`:
   - `outputSuccess<T>(data: T): void` -- writes JSON envelope to stdout via `process.stdout.write()`
   - `outputError(error: string, code?: string): void` -- writes error envelope to stdout
   - `logStatus(message: string, quiet?: boolean): void` -- writes to stderr if not quiet
   - Import types from `./types.js` (note `.js` extension for ESM)
   - NEVER use console.log -- always process.stdout.write or process.stderr.write

3. Create `src/lib/errors.ts`:
   - Base class `TgError extends Error` with `code` property
   - `CredentialError extends TgError` -- for missing/invalid API credentials
   - `SessionError extends TgError` -- for session-related errors
   - `FloodWaitError extends TgError` with `seconds` property -- for rate limiting
   - Helper `formatError(err: unknown): { message: string, code?: string }` -- extracts error info from any thrown value

4. Create `src/lib/config.ts`:
   - `createConfig(configPath?: string): Conf<TgConfig>` -- creates conf instance with projectName "tg-cli", schema for apiId (number), apiHash (string), profiles (object), default profiles: {}
   - `resolveCredentials(config: Conf<TgConfig>): { apiId: number, apiHash: string } | null` -- env vars TG_API_ID + TG_API_HASH first, then config file, returns null if neither
   - `getCredentialsOrThrow(config: Conf<TgConfig>): { apiId: number, apiHash: string }` -- calls resolveCredentials, throws CredentialError with helpful message including https://my.telegram.org/apps link if null

5. Create `src/lib/prompt.ts`:
   - `createPrompt(): { ask(question: string): Promise<string>, close(): void }` -- wraps readline/promises createInterface with input: process.stdin, output: process.stderr (prompts go to stderr, not stdout)
   - The returned `ask` function calls rl.question()
   - The returned `close` function calls rl.close()

6. Write tests FIRST (TDD red phase), then implement:
   - `tests/unit/output.test.ts`: Test outputSuccess envelope structure, outputError with/without code, logStatus to stderr, logStatus quiet mode. Mock process.stdout.write and process.stderr.write using vi.spyOn.
   - `tests/unit/config.test.ts`: Test resolveCredentials with env vars set, with config values, with neither. Use vi.stubEnv for env var testing. Test createConfig returns a Conf instance.

7. Run tests, verify all green.
  </action>

  </verify>
