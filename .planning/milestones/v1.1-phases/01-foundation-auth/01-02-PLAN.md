  <name>Task 1: Session store, client wrapper, and rate limiting</name>
  <files>src/lib/session-store.ts, src/lib/client.ts, src/lib/rate-limit.ts, tests/unit/session-store.test.ts, tests/unit/client.test.ts, tests/unit/rate-limit.test.ts</files>
  <behavior>
    - SessionStore.save("default", "abc123") creates sessions directory and writes "abc123" to sessions/default.session file
    - SessionStore.load("default") returns "abc123" from the file (returns empty string if file doesn't exist)
    - SessionStore.delete("default") removes the session file
    - SessionStore uses proper-lockfile to lock session files during read and write
    - withClient() calls client.connect(), runs the callback, then calls client.destroy() in finally block
    - withClient() clears the safety timeout after successful completion
    - withClient() sets a 30-second safety timeout that calls process.exit(1) if cleanup hangs
    - withRateLimit wraps an async function and catches FloodWaitError from gramjs
    - withRateLimit for FloodWait under threshold: gramjs handles automatically (no action needed)
    - withRateLimit for FloodWait above threshold: returns structured error with seconds to wait
  </behavior>
  <action>
1. Create `src/lib/session-store.ts`:
   - Class `SessionStore` with constructor taking `configDir: string`
   - Creates `sessions` subdirectory under configDir on construction (mkdirSync recursive)
   - `async load(profile: string): Promise<string>` -- reads session file, returns empty string if not found. Uses proper-lockfile lock/unlock for file access.
   - `async save(profile: string, sessionString: string): Promise<void>` -- writes session string to file with lock. Creates empty file first if needed (proper-lockfile requires existing file).
   - `async delete(profile: string): Promise<void>` -- removes session file if exists
   - `filePath(profile: string): string` -- returns `join(dir, 'sessions', `${profile}.session`)`
   - Lock options: `{ retries: { retries: 3, minTimeout: 100 } }`

2. Create `src/lib/rate-limit.ts`:
   - `async function withRateLimit<T>(fn: () => Promise<T>, label?: string): Promise<T>` -- wraps async call, catches errors. If error has `seconds` property (FloodWaitError from gramjs), log wait time to stderr and throw a structured FloodWaitError from errors.ts with the seconds value. All other errors rethrow as-is.
   - The gramjs client already handles FloodWait under `floodSleepThreshold` (60s default) by sleeping automatically. This wrapper handles the case where it throws for waits ABOVE threshold.

3. Create `src/lib/client.ts`:
   - Import TelegramClient from 'telegram', StringSession from 'telegram/sessions'
   - Interface `ClientOptions { apiId: number, apiHash: string, sessionString: string }`
   - `async function withClient<T>(opts: ClientOptions, fn: (client: TelegramClient) => Promise<T>): Promise<T>`:
     - Create TelegramClient with StringSession, connectionRetries: 3, retryDelay: 1000, floodSleepThreshold: 60
     - Set 30-second safety timeout that destroys client and calls process.exit(1)
     - In try: connect(), run fn(client), return result
     - In finally: clearTimeout, await client.destroy().catch(() => {})
     - Use `destroy()` NOT `disconnect()` (per research -- avoids zombie _updateLoop)
   - `async function createClientForAuth(apiId: number, apiHash: string): Promise<TelegramClient>`:
     - Creates client with empty StringSession (for fresh login)
     - Does NOT connect or set timeout -- caller manages lifecycle for interactive auth flow
     - Returns the client instance for use with client.start()

4. Write tests FIRST (red), then implement:
   - `tests/unit/session-store.test.ts`: Use tmp directory (os.tmpdir + randomUUID). Test save/load/delete cycle. Test load returns empty string for missing profile. Test concurrent access doesn't throw (two sequential save/load with same profile). No need to test actual locking mechanism in unit tests -- just that save/load/delete work correctly.
   - `tests/unit/client.test.ts`: Mock TelegramClient from 'telegram'. Test withClient calls connect then destroy. Test withClient calls destroy even if fn throws. Test safety timeout is set and cleared. Use vi.mock('telegram') to mock the client.
   - `tests/unit/rate-limit.test.ts`: Test successful call passes through. Test FloodWaitError (with `seconds` property) is caught and rethrown as structured FloodWaitError. Test non-flood errors are rethrown as-is.
  </action>

   - Create config via createConfig, get credentials via getCredentialsOrThrow
   - Create prompt via createPrompt() (from src/lib/prompt.ts)
   - Create client via createClientForAuth(apiId, apiHash)
   - Call `client.start()` with callbacks:
     - `phoneNumber`: `async () => prompt.ask('Phone number (international format): ')`
     - `phoneCode`: `async (isCodeViaApp) => prompt.ask(isCodeViaApp ? 'Code (from Telegram app): ' : 'Code (from SMS): ')`
     - `password`: `async (hint) => prompt.ask(hint ? `2FA password (hint: ${hint}): ` : '2FA password: ')`
     - `onError`: log to stderr
   - After start(): get session string via `client.session.save() as unknown as string`

   - Output success with `{ session: sessionString.substring(0, 20) + '...', phone }`
   - In finally: destroy client, close prompt

2. Create `src/commands/auth/status.ts`:
   - Export `async function statusAction(this: Command)`
   - Get global opts, create config and SessionStore
   - Load session string for profile

   - If session exists: use withClient to connect and call `client.checkAuthorization()`
   - If authorized: get user info via `client.getMe()`, output `{ authorized: true, user: { id, phone, username, firstName } }`
   - If not authorized: output `{ authorized: false, reason: 'Session expired or invalid' }`

   - Output success: `{ loggedOut: true }`
